# WIO (Work In One) — 마스터 아키텍처 문서
## Ten:One™ Universe 업무 운영 플랫폼 · 완전 설계서

> "시스템이 알아서 일한다. 사람은 확인하고 판단만 한다."
> 이 문서는 WIO의 비전·설계·개발 원칙·기능·파이프라인을 담은 단일 진실 소스다.

## PART 1 — 왜 WIO인가

### 1-1. 문제 정의

기업들은 평균 5~7개 툴을 동시에 쓴다.
슬랙으로 소통하고, 노션에 문서 쓰고, 지라로 프로젝트 관리하고,
세일즈포스로 영업하고, 엑셀로 재무 정리한다.

결과: 정보가 파편화된다. 툴 비용이 쌓인다. 사람이 툴에 맞춰 일한다.

노션과의 근본적 차이:
- 노션: "사람이 넣고 AI가 정리"
- **WIO: "AI가 넣고 사람이 확인"**

### 1-2. WIO의 포지셔닝

```
노션 + 슬랙 + 세일즈포스 + 탈렌트LMS + 커뮤니티 플랫폼
= 월 수백만원 × 여러 툴

WIO = 이걸 하나로.
      한국 시장 최적화.
      AI 내장.
      커뮤니티·교육·인재까지.
      마케팅·광고·크리에이티브 업계 특화.
```

### 1-3. 핵심 원칙

```
입력을 없앤다.
입력이 불가피하면 AI가 80% 채워준다.
사람은 연결하고 판단하고 성장한다.
```

## PART 2 — 시스템 구조

### 2-1. 3계층 구조

```
WIO (기능 공장)
  └── 27개 모듈을 만들고 지속적으로 업데이트한다

TenOne (컨트롤타워)
  └── super_admin으로 Universe 23개 브랜드 종합 관리

각 브랜드 (역할 담당자)
  └── brand_id + RLS로 격리·공유
      필요한 모듈 조합만 탑재
```

### 2-2. 설계 원칙 5가지 — 이걸 어기면 전체가 무너진다

**원칙 1: Single Source of Truth**
```sql
-- 모든 데이터는 하나의 Supabase DB에 있다
-- 같은 데이터를 두 곳에 저장하지 않는다
-- 브랜드별 추가 정보는 profiles 테이블에 brand_id로 확장

✅ users 테이블 하나 → 모든 브랜드가 참조
❌ madleague_users + badak_users 따로 만들지 말 것
```

**원칙 2: 멀티테넌트 — RLS 기반**
```
단일 Supabase DB + Row Level Security
테넌트 = 브랜드 (brand_id)
모든 테이블에 brand_id 컬럼 필수
TenOne super_admin은 brand_id 없이 전체 접근 가능
스키마 분리(별도 DB) 절대 금지 → 브랜드 간 연결이 불가능해짐
```

**원칙 3: 모듈 독립성**
```
각 모듈은 다른 모듈 없이도 작동한다
모듈 간 연동은 이벤트(webhook) 또는 API 호출로만
직접 DB JOIN으로 모듈 간 의존성 금지

✅ Learn 수료 이벤트 → People API 호출 → 뱃지 부여
❌ Learn 쿼리에서 people 테이블 직접 JOIN
```

**원칙 4: 역할 체계 통일**
```typescript
// 시스템 레벨
system_role: 'super_admin' | 'brand_admin' | 'member' | 'guest'

// 브랜드 내 역할 (각 브랜드가 커스텀)
brand_roles: {
  madleague: 'leaguer' | 'mentor' | 'staff' | 'admin',
  badak: 'member' | 'mentor' | 'staff' | 'admin',
  hero: 'talent' | 'consultant' | 'client' | 'admin',
  smarcomm: 'crew' | 'pm' | 'client' | 'admin',
  // ...
}
// users 테이블: system_role + brand_roles JSON
```

**원칙 5: API 응답 구조 통일**
```typescript
interface APIResponse<T> {
  success: boolean
  data: T | null
  error: string | null
  meta?: { total?: number; page?: number; brand_id?: string }
}

// 엔드포인트 패턴
GET    /api/[module]/[resource]
POST   /api/[module]/[resource]
PUT    /api/[module]/[resource]/[id]
DELETE /api/[module]/[resource]/[id]
```

### 2-3. 핵심 DB 스키마

```sql
-- 브랜드 마스터
CREATE TABLE brands (
  id        text PRIMARY KEY,  -- 'madleague', 'badak', 'hero' 등
  name      text,
  status    text,              -- 'active' | 'dev' | 'incubating'
  modules   jsonb,             -- 활성화된 모듈 목록
  settings  jsonb,             -- 브랜드별 설정 (테마, 기능 on/off)
  created_at timestamptz
);

-- 통합 회원 (Universe 전체 공유 — 한 번 가입으로 전체 접근)
CREATE TABLE users (
  id           uuid PRIMARY KEY,
  email        text UNIQUE,
  name         text,
  avatar_url   text,
  system_role  text DEFAULT 'member',
  brand_roles  jsonb DEFAULT '{}',
  points       int DEFAULT 0,
  grade        text DEFAULT 'seed', -- seed→sprout→grow→pro→master
  created_at   timestamptz
);

-- 브랜드별 프로필 (확장 정보)
CREATE TABLE profiles (
  id           uuid PRIMARY KEY,
  user_id      uuid REFERENCES users(id),
  brand_id     text REFERENCES brands(id),
  bio          text,
  tags         text[],
  custom_data  jsonb,  -- 브랜드별 추가 필드
  created_at   timestamptz,
  UNIQUE(user_id, brand_id)
);

-- 테이블 네이밍 규칙: [module]_[resource]
-- 모든 테이블에 brand_id 포함 (필수)
-- 예: project_projects, learn_courses, talk_posts
```

## PART 5 — AI 파이프라인 (TrendHunter 통합 설계)

### 5-1. AI 입력 3경로

```
경로 1: 완전 자동 (사람 개입 0%)
  └── TrendHunter 크롤링, 이메일/카톡 패턴 감지,
      캘린더 동기화, Git 커밋 추적,
      프로젝트 완료→정산 트리거 체인

경로 2: AI 초안 → 사람 확인 (AI 80% + 사람 20%)
  └── 프로젝트 등록, 회의록→액션 아이템,
      시수표, 정산서, 콘텐츠 초안, 주간 리포트,
      인재 매칭 추천

경로 3: 간편 입력 (불가피할 때, 최소한으로)
  └── 카톡/슬랙 자연어 → AI 구조화
      사진(영수증/명함/계약서) → OCR + AI 파싱
      음성 메모 → 텍스트 변환 + 분류
```

### 5-2. TrendHunter 크롤러 체계

```
수집층 (Collectors)
├── 🔩 바닥쇠 — 카카오 오픈채팅방 크롤러
│   ├── 내부 14개 방: 응답 모드 (수집 + 바닥쇠 호출 시 AI 응답)
│   ├── 공지방: 수집만 모드
│   └── 외부 N개 방: 인간 모드 (조용히 수집만)
│   기술: 안드로이드 공기계 + 메신저봇R
│
├── 🌐 웹크롤러 — 커뮤니티·정보 사이트
│   네이버 카페, 블라인드, 에펨코리아, 클리앙, 디스콰이엇
│   기술: Puppeteer (헤드리스 브라우저)
│
├── 💬 디스코드 크롤러
│   마케팅/AI/크리에이티브 서버
│   기술: Discord.js
│
├── 📰 뉴스·RSS·뉴스레터 크롤러
│   구독 뉴스레터(Gmail API), RSS 피드, 뉴스 사이트
│   기술: RSS 파서 + Gmail API
│
└── 🏛 SmarComm 크롤러 (공공 데이터)
    나라장터, bizinfo, 공모전 사이트

분석층 (Claude 기반)
├── 키워드 추출 + 트렌드 감지
├── 사업 기회 패턴 매칭 (인재수요→HeRo / 솔루션→SmarComm 등)
├── 커뮤니티 헬스 분석
└── 콘텐츠 초안 자동 생성

활용층 (Output)
├── tenone.biz/trendhunter 사이트
├── 1 신호 → 6개 브랜드 콘텐츠 자동 변환
├── 일간 기회 알림 → 텐원
└── 커뮤니티 헬스 리포트
```

### 5-3. 1 신호 → 다중 변환 파이프라인


## PART 11 — UX 설계 원칙

### 11-1. 모듈별 아하 모멘트 정의

사람이 툴에 빠지는 건 기능이 많아서가 아니다. 한 순간의 경험 때문이다.
각 모듈에 아하 모멘트를 반드시 설계한다.

```
Project   — 태스크 하나 완료 시 프로젝트 손익이 실시간으로 바뀌는 걸 처음 보는 순간
People    — 내 프로필에 "이 사람과 공통 연결 3명"이 뜨는 순간
GPR       — 내 주간 달성률이 팀 평균보다 높다는 걸 시각적으로 확인하는 순간
Learn     — 수료증 PDF가 자동으로 생성되는 순간
Timesheet — 월요일 아침에 지난주 시수표가 이미 80% 채워져 있는 순간
Talk      — AI가 어제 게시글 10개를 3줄로 요약해서 보여주는 순간
Finance   — 이번 달 손익이 "지난달 대비 +23%"로 한 화면에 보이는 순간
```

### 11-2. 피로감 제거 3원칙

**원칙 1: 빈 화면이 없다**
```
"아직 데이터가 없습니다" → 금지
대신: "이렇게 시작해보세요" + 예시 + 시작 버튼

모든 Empty State 구성:
- 무엇을 할 수 있는지 한 문장 설명
- 실제 사용 예시 (스크린샷 or 일러스트)
- 시작 CTA 버튼 1개
```

**원칙 2: 칭찬이 있다**
```
프로젝트 완료 시   → 작은 축하 애니메이션
GPR 목표 달성 시  → 뱃지 자동 부여
월간 활동 리뷰    → "이번 달 태스크 23개 완료!"
처음 모듈 사용 시 → "첫 번째 프로젝트 생성 완료"
작은 피드백이 계속 쓰게 만든다
```

**원칙 3: 기억해준다**
```
마지막 사용 위치 복원
"저번에 하던 거 이어서 하시겠어요?"
AI가 내 패턴 학습 ("매주 월요일 GPR 작성하시죠?")
자주 쓰는 기능이 홈 대시보드에 자동으로 올라온다
```

### 11-3. 모듈 취사선택 구조 — 데이터 끊김 없이

**핵심 원칙: 모듈이 없어도 데이터는 살아있다**

```
Timesheet 모듈을 해지해도
  → 시수 데이터는 DB에 보존 (삭제 아님)
  → Project 모듈에서 "투입 시간" 형태로 계속 보인다
  → 다시 켜면 이력이 그대로 복원된다

Finance 모듈이 없어도
  → Project에서 예산/비용 입력은 가능하다
  → Finance를 켜는 순간 입력 데이터가 손익계산서로 즉시 연결된다
```

**3단계 레이어**

```
Level 1: 모듈 OFF
  사이드바 UI 사라짐 / 데이터 DB 보존 / 다른 모듈에서 요약만 표시

Level 2: 모듈 ON (기본)
  핵심 기능만 활성화 / UI 단순

Level 3: 모듈 ON (확장)
  고급 기능 추가 / 세부 기능 토글 가능
```

**Feature Toggle — 모듈 안 취사선택**

```
brands.settings.modules[n].features[] 배열로 관리

Project 모듈:
  ✅ 칸반 보드         (기본 ON, 끄기 불가)
  ✅ 태스크 관리       (기본 ON, 끄기 불가)
  ☐  간트 차트        (선택 OFF 가능)
  ☐  예산/손익        (Finance 연동 시 자동 ON)
  ☐  타임시트 연동     (Timesheet 있을 때만 표시)
  ☐  AI 일정 추천     (AI 모듈 있을 때만 표시)

Talk 모듈:
  ✅ 게시판/공지       (기본 ON, 끄기 불가)
  ✅ 이벤트            (기본 ON)
  ☐  채팅             (선택 OFF 가능 — 슬랙 병행 조직)
  ☐  투표/설문        (선택 OFF 가능)
  ☐  AI 요약          (AI 모듈 있을 때만 표시)
```

**Bridge 필드 — 연동 모듈 없을 때 폴백**

```
Timesheet 없는 조직의 Project:
  → "투입 시간 직접 입력" 필드 자동 생성
  → Timesheet 추가 시 이 필드가 자동 연동으로 전환
  → 사용자는 변화를 거의 느끼지 못한다

Finance 없는 조직의 Project:
  → 예산·비용 입력 가능, 손익계산서 없음
  → Finance 추가 시 기존 입력 데이터가 즉시 손익표로 연결

AI 없는 조직의 Timesheet:
  → 수동 입력 그리드
  → AI Assist 추가 시 캘린더 연동 자동 추천으로 전환
```

## PART 12 — 재무·회계 설계

### 12-1. 프로젝트 실시간 손익 계산식

```
매출총이익 = 프로젝트 계약금액 - (투입 시수 × 시급)
영업이익   = 매출총이익 - 프로젝트 고정비(경비·소프트웨어 등)
영업이익률 = 영업이익 ÷ 프로젝트 계약금액 × 100

경보 트리거 (자동 알림):
- 영업이익률 < 20%  → PM에게 황색 경고
- 영업이익률 < 0%   → 적자 전환, 적색 경보 + TenOne 알림
- 투입 시수 > 예산의 80%  → "시수 초과 위험" 알림
- 미청구 완료 프로젝트    → "청구서 발행 필요" 자동 알림
```

### 12-2. Timesheet → ERP 실시간 파이프라인

```
시수 입력 (Timesheet)
  → PM 승인 확정
  → 프로젝트 In-Cost 자동 업데이트 (Project)
  → Finance 손익계산서 실시간 반영
  → Insight Gap 분석 자동 갱신
     (예산 시수 vs 실투입 시수 / 예산 비용 vs 실비용)
  → 경보 트리거 체크
  → Universe Dashboard 전사 손익 집계
```

### 12-3. 현금흐름(Cash Flow) 추적

```
흑자도산 방지 원칙:
손익이 플러스여도 현금이 없으면 회사가 멈춘다
손익과 현금흐름을 반드시 분리 표시한다

미수금 추적:
- 청구 발행일 → 입금 확인일까지 D+N일 자동 계산
- 매출채권 연령 분석: 30일/60일/90일 이상 미수금
- 입금 지연 → 담당자 팔로업 트리거 자동 발송

월별 현금 예측:
- 이번 달 예정 입금 (계약 완료 건)
- 이번 달 예정 지출 (크루 정산·경비)
```

### 12-4. 원가율 벤치마크

```
프로젝트 유형별 목표 원가율:
- 마케팅 대행: 인건비율 40% 이하
- 브랜딩 컨설팅: 인건비율 35% 이하
- 교육 프로그램: 인건비율 50% 이하
```

### 12-5. 세금·규정 자동화

```
크루 정산 시:
```

## PART 14 — Universe 내부 운영 구조
### "TenOne이 WIO의 첫 번째 고객이다"

> Ten:One™ Universe 23개 브랜드는 개별 고객이다.
> TenOne은 WIO를 직접 쓰면서 만든다. 우리가 불편하면 고객도 불편하다.
> 이것이 도그푸딩(Dogfooding)이자 가장 강력한 세일즈다.

### 14-1. 3계층 관리 구조

```
Level 0: WIO Platform Admin (TenOne super_admin)
  └── 전체 플랫폼 인프라 관리
      모든 테넌트(브랜드) 생성·수정·삭제
      전체 DB·서버·비용 모니터링
      Universe Dashboard로 23개 브랜드 한눈에

Level 1: Brand Admin (각 브랜드 운영자)
  └── 자기 브랜드 테넌트만 접근
      모듈 ON/OFF, Feature Toggle 설정
      멤버 초대·역할 배정
      브랜드 테마(컬러·로고) 설정

Level 2: Brand Member (각 브랜드 구성원)
  └── 배정된 모듈만 사용
      자기 역할(brand_role)에 따른 기능 접근
```

### 14-2. TenOne이 각 브랜드를 관리하는 방식

**기술적 구조:**
```sql
-- TenOne super_admin은 brand_id 없이 전체 조회
-- 일반 brand_admin은 자기 brand_id만 조회 (RLS)

-- 테넌트 생성 플로우 (신규 브랜드 온보딩)
1. TenOne super_admin이 brands 테이블에 브랜드 등록
2. brand_id 자동 생성 (예: 'madleague', 'badak')
3. 브랜드별 modules[] 배열 설정 (어떤 모듈 사용할지)
4. 첫 번째 brand_admin 계정 생성 + 초대 이메일
5. brand_admin이 멤버 초대 시작
```

**운영 주기:**
```
실시간: Universe Dashboard 이상치 감지 알림
주간:   브랜드별 활동 리포트 자동 생성 → TenOne 확인
월간:   수익 현황, 인재 파이프라인 전환율, 헬스스코어 리뷰
분기:   GPR 캐스케이딩 — TenOne 전사 목표 → 브랜드 목표 → 개인
```

### 14-3. 브랜드별 모듈 설정 관리

**Brand Admin이 할 수 있는 것:**
```
브랜드 설정 페이지 (/app/settings):

[기본 설정]
- 브랜드명, 로고, 대표색상, 도메인
- 기본 언어, 시간대, 날짜 형식

[모듈 관리]
- 활성화된 모듈 목록 확인
- 모듈 안 Feature Toggle 켜고 끄기
- 업그레이드 필요 모듈 → TenOne에 요청

[멤버 관리]
- 초대 링크 생성 / 이메일 초대
- 역할 배정 (brand_role 커스텀)
- 비활성 멤버 관리

[연동 설정]
- 슬랙/구글 캘린더/깃헙 등 외부 연동 on/off
- 웹훅 설정 (이벤트 발생 시 외부 알림)
- API 키 발급 (외부 시스템 연동용)
```

**TenOne만 할 수 있는 것 (super_admin 전용):**
```
- 브랜드 신규 생성 / 삭제
- 구독 플랜 변경 (어느 모듈까지 허용할지)
- 크로스 브랜드 데이터 조회 (전체 Universe)
- 브랜드 간 데이터 연결 승인
  (예: MADLeague 리거 → HeRo 인재 등록 동의 프로세스)
- 전사 GPR 목표 설정 → 브랜드로 캐스케이딩
- 전체 AI 사용량·비용 모니터링
```

### 14-4. 브랜드 간 데이터 연결 프로세스

```
MADLeague 우수 리거 → HeRo 인재 등록:

1. MADLeague admin이 "HeRo 추천" 버튼 클릭
2. 해당 리거에게 동의 요청 알림 발송
3. 리거가 동의 → HeRo 인재 프로필 자동 생성
4. TenOne Universe Dashboard에 "파이프라인 전환" 기록
5. HeRo admin에게 신규 인재 알림

거절하거나 동의 취소 시:
- HeRo 프로필 즉시 삭제
- MADLeague 쪽 데이터에 "HeRo 비연동" 표시
- 재요청은 30일 후 가능
```

```
  → 월간 리뷰 미팅 (정기 지원)
```

**모델 C: Partner Sales (파트너 대행 판매)**
```
타겟: WIO Certified Partner가 고객사에 도입
경로: 파트너가 고객 발굴 → WIO 플랫폼에서 테넌트 생성 대행

파트너 조건:
  - WIO Certified 교육 이수
  - 최소 2개 고객사 도입 이력
  - 파트너 포털 계정 보유

파트너 수익:
  - 첫 1년 계약금의 20~30% 리베이트
  - 파트너 포털에서 실시간 정산 확인

자연 파트너 풀:
  Badak 현업 (에이전시 대표·팀장) → 자기 회사에 WIO 도입
  SmarComm 클라이언트 → 마케팅 대행 받으면서 WIO도 도입
```

### 15-2. 클라이언트 온보딩 5단계 (5F)

```
Find → Form → Fire → Fruit → Feed

1. Find (발견) — 현황 진단
   - 현재 사용 중인 툴 목록 + 비용 계산
   - 주요 Pain Point 3가지 도출
   - 필요 모듈 구성 설계
   산출물: WIO 도입 제안서 (자동 생성)

2. Form (설계) — 테넌트 구성
   - 테넌트 생성 (brand_id, 도메인, 테마)
   - 모듈 ON/OFF + Feature Toggle 설정
   - 역할 체계 설계 (brand_role 커스텀)
   - 데이터 이관 계획
   산출물: WIO 세팅 완료 체크리스트

3. Fire (실행) — 전환 실행
   - 기존 데이터 CSV → WIO 임포트
   - 구성원 이메일 초대 (일괄 발송)
   - 관리자 1:1 교육 (2시간)
   - 전체 구성원 온보딩 세션 (1시간)
   산출물: 온보딩 완료 리포트

4. Fruit (성과) — 1개월 성과 확인
   - 헬스스코어 측정 (목표: 70점 이상)
   - 이전 대비 업무 효율 지표 수집
   - 추가 필요 모듈 발굴
   산출물: 1개월 성과 리포트

5. Feed (피드백) — 지속 개선
   - 월간 리뷰 미팅 (30분)
   - 새 기능 업데이트 안내
   - 확장 모듈 제안
   산출물: 분기별 최적화 제안서
```

### 19-1. "완벽"의 정의

완벽이란 기능이 많은 게 아니다. 이 7가지를 전부 통과한 모듈만 출시한다.

```
□ 모든 기능이 실제 DB에 저장된다
□ 에러가 없다
□ 모바일에서 완벽히 작동한다
□ 보안이 통과된다 (RLS, 권한 체계, 파일 보안)
□ Universe 내부에서 실전 사용해서 검증됐다
□ 처음 쓰는 사람이 설명 없이도 쓸 수 있다
□ 이 모듈만 팔아도 돈을 받을 수 있는 수준이다
```

### 19-3. Talk 완성 체크리스트 (첫 번째 출시 기준)

**게시판:**
```
□ 글 작성·수정·삭제 DB 저장
□ 댓글·대댓글 (2depth) 작동
□ 이미지 업로드 → Supabase Storage → CDN
□ 멘션(@이름) → 해당 멤버 알림
□ 좋아요·북마크
□ 관리자 삭제·숨김·이동
□ 신고 시스템 (3회 자동 숨김)
□ 풀텍스트 검색
□ 카테고리·태그
□ 임시 저장 자동 (30초)
□ 모바일 완벽 작동
□ 빈 상태 가이드 화면
```

**채팅:**
```
□ 1:1 DM 실시간
□ 그룹 채팅
□ 채널 (공개·비공개)
□ 타이핑 인디케이터
□ 읽음 확인 (DM)
□ 메시지 수정·삭제
□ 파일·이미지 공유
□ 푸시 알림 (PWA)
□ 알림 설정 (채널별 on/off)
□ 방해 금지 시간대
□ 모바일 완벽 작동
```

**보안:**
```
□ RLS brand_id 완전 격리
□ 파일 업로드 바이러스 스캔
□ Rate Limiting (1분당 게시글 5개, 댓글 20개)

□ 1:1 DM 관리자 열람 불가
□ 익명 게시판 user_id 암호화 저장
```

**검증 기준:**
```
□ Badak 실전 1주일 사용

```

### 19-4. 출시 후 검증 → 다음 모듈 원칙

