# Part I. 시스템 기반

---

## 1. 시스템 개요

### 1.1 목적

거대 기업의 모든 부서가 하나의 플랫폼 위에서 업무를 수행하되, 각 조직은 자기 업무에 최적화된 모듈만 사용하는 **모듈러 ERP 시스템**. 기존 ERP가 "기능 중심"으로 설계되었다면, 이 시스템은 **"조직 중심 + 모듈 조합"** 방식이다.

### 1.2 설계 원칙

1. **조직이 먼저, 기능은 그 다음**: 트랙 → 조직 → 인력 → 권한 → 모듈 순서로 세팅
2. **3대 자원 중심**: 모든 모듈은 인간/시간/돈 중 하나 이상을 입출력으로 가짐
3. **모듈 독립성**: 각 모듈은 독립 배포·업데이트 가능 (마이크로서비스)
4. **조직 간 워크플로우**: 부서를 넘나드는 업무 프로세스를 시각적으로 설계
5. **권한 계층화**: 시스템 관리자 → 트랙 관리자 → 조직 관리자 → 일반 사용자
6. **확장 가능성**: 모듈 카탈로그에 신규 모듈을 플러그인처럼 추가
7. **문화 내재화**: 기업 철학·가치가 양식·프로세스·평가에 구조적으로 반영
8. **데이터 중심 경영**: 내외부 모든 데이터를 수집·통합·분석하여 의사결정 반영

### 1.3 WIO 원칙

| 원칙 | 적용 |
|------|------|
| 입력을 없앤다 | 매체 API 자동 수집, 근태 자동 기록, 전표 자동 생성, AI 문서 초안 |
| AI가 80% 채운다 | 감성 분석, 어트리뷰션 자동 계산, 가치 정합성 체크 |
| 사람은 20%의 판단 | 전략 방향, 최종 승인, 핵심 가치 판단 |

### 1.4 핵심 개념 8가지

| 개념 | 비유 | 정의 |
|------|------|------|
| **트랙** | 건물의 층 | 7개 대분류 업무 영역 |
| **조직** | 층 안의 방 | 실제 부서·팀·파트 |
| **모듈** | 방의 가구 | 카탈로그에서 골라 배정하는 독립 기능 단위 |
| **워크플로우** | 동선/복도 | 업무가 흘러가는 순서·규칙·조건 |
| **도구** | 문구/장비 | 일상 업무용 생산성·협업 소프트웨어 |
| **권한** | 출입 카드 | 접근 범위를 결정하는 규칙 |
| **양식** | 표준 서류함 | 철학이 내장된 공식 문서 서식 |
| **대시보드** | 안내 전광판 | 역할별 데이터 시각화 화면 |

**세팅 순서:** 트랙 생성 → 조직 배치 → 인력 배치 → 권한 설정 → 모듈 배정 → 전사 워크플로우 → 부서 워크플로우 → 도구 활성화 → 개인 대시보드 자동 제공

**누가 세팅하는가:**
- 시스템관리자 → 트랙·조직·권한·양식 (구조)
- 경영기획 → 전사 워크플로우 (강제)
- 부서장 → 부서 워크플로우 + 모듈 + 도구 (자율)
- 개인 → 개인 도구 + 대시보드 (자동 제공)

## 2. 3대 자원 관리 체계

모든 모듈이 공유하는 코어 데이터 레이어.

### 인간 (Human)
인사 마스터(사원번호·부서·직급), 조직 소속(다중 허용), 역량 프로필(스킬·자격·교육·프로젝트), 근태(출퇴근·휴가·초과), 성과(KPI·평가·피드백), 권한 프로필(RBAC)

### 시간 (Time)
근무 시간(자동 집계·초과 계산), 프로젝트 타임라인(마일스톤), 업무 일정(개인/팀/부서), SLA/납기, 워크플로우 리드타임

### 돈 (Money)
회계 원장(GL·분개·전표·재무제표), 예산(편성·집행·잔액), 매출/매입(파이프라인·세금계산서), 급여(4대보험·원천징수), 비용(경비·법인카드·출장비), 자산(감가상각·재고)

## 3. 기업 철학의 시스템 내재화

### 3.1 왜 시스템에 녹이나

문화는 포스터가 아니라 **양식·프로세스·결재·피드백 구조 안에 녹아** 있어야 체화된다. "이렇게 일해라"가 아니라 "이렇게 일할 수밖에 없는 구조"를 만드는 것.

### 3.2 Culture Engine

시스템 관리자가 미션·비전·핵심가치(3~7개)·일하는 원칙·커뮤니케이션 가이드를 등록하면, 양식·프로세스·평가 전반에 자동 반영되는 엔진.

> Claude Code 가이드: `CorporateCulture` → `CoreValue`(행동 지표+반례) → `WorkPrinciple`(적용 대상별) → `CommunicationGuide`(보고/회의/외부별 톤). 버전 관리, 활성/비활성 상태 포함.

### 3.3 양식 시스템 — 철학이 녹은 문서

| 양식 | 핵심 반영 |
|------|----------|
| 기안서 | "어떤 핵심가치에 기여하는가" 필수. 기대효과를 3대 자원 관점에서 |
| 보고서 | "결론→근거→제안→리스크" 순서 고정. 대상별 상세도 자동 조절 |
| 제안서 | 고객 가치·기술 실현성·재무 임팩트 3축 내장 |
| 회의록 | 유형별 구조 분리. 결정사항+후속+담당+기한 필수 |
| 피드백 | 핵심가치 기반 구조 (어떤 가치에 대한 피드백인지 선택) |

> Claude Code 가이드: `DocumentTemplate` → `TemplateSection`(필드타입·필수여부) → `CultureLink`(양식↔가치 연결) → `AudienceVariant`(대상별 다른 구성). AI 리뷰 기준도 양식에 내장.

### 3.4 프로세스에 녹이는 방법

**결재**: 기안→팀장→가치 정합성 체크(AI 스코어)→본부장
**회의**: 유형별 강제(의사결정=투표+태스크 전환, 스탠드업=3문장+15분 타이머, 1:1=피드백 기록)
**온보딩**: Day1 미션/비전 → Day2 핵심가치 사례 → Day3 양식 작성(AI 코칭) → Week2 동료 피드백
**AI 코칭**: 문서 작성 시 실시간 가이드 ("결론이 3줄 초과", "고객 가치가 불명확", "수치 누락")

### 3.5 Culture Dashboard

가치 언급 빈도, 양식 준수율(AI 판정), 회의 효율성(결정 전환율), 피드백 활성도, 온보딩 완주율, 가치 정합 스코어

### 4.1 7개 트랙

| 트랙 | 조직 | 운영 |
|------|------|------|

| **T2 사업** | 마케팅본부(전략/브랜드/퍼포먼스/콘텐츠·소셜/CRM마케팅/데이터/운영), 영업(국내/해외/기술), 사업개발(BD), 고객관리(CS/CX) | 전문 부서 |
| **T3 생산** | 생산관리(MPS/MRP), 공정(MES), 품질(QC), 설비, 구매·조달, 물류·유통(WMS/TMS/SCM) | 전문 부서 |
| **T4 지원** | 연구소(R&D), 개발(프론트/백엔드/인프라/QA), 디자인(UI·UX/BX·CI/영상), 데이터(엔지니어링/분석/AI·ML) | 전문 부서 |
| **T5 파트너** | 내부 파트너(계열사), 외부 파트너(협력사·벤더), 컨설팅/어드바이저, 프리랜서 풀 | 전문 부서 |
| **T6 공통** | My Dashboard(인사·평가·업무·기안), 소통(메일·메신저·게시판), 협업(캘린더·결재·프로젝트), 피드백·인정, AI비서 | **전 직원** |
| **T7 시스템관리** | 사용자·권한·조직·모듈·워크플로우 관리, 전산·장비, 모니터링·보안·감사로그, Culture Engine | IT/전산 부서 |

### 4.2 트랙 운영 원칙

| 구분 | 트랙 | 사용자 | 관리 주체 |
|------|------|--------|----------|
| 전문 | T1~T5 | 해당 부서 직원 | 부서장 |
| 공통 | T6 | **전 직원** | 인사/총무 |
| 관리 | T7 | IT/전산 직원 | CIO/IT팀장 |

**Track 6 핵심**: 각 전문 트랙(T1~T5)에서 관리되는 데이터가 Track 6 대시보드에 **개인 기준**으로 집계되어 보인다. 인사팀이 평가를 관리하지만(T1), 직원 본인은 자기 평가를 T6에서 확인한다.

### 4.3 지주사 수준 추가 조직

지주사(Holding) 아래: 통합 마케팅 인텔리전스 센터(미디어 전략/데이터 분석/브랜드/고객 인사이트/거버넌스) + 데이터전략실(CDO Office). 각 사업회사(BU)는 7개 트랙 구조를 자체 보유.

### 4.4 조직 데이터 모델

> Claude Code 가이드: `Organization`(id, name, code, track_id, parent_org_id, level[본부/팀/파트/셀], head_user_id, status) + `Track`(id, name, code, sort_order, admin_user_ids)

## 5. 시스템 세팅 플로우

| Step | 내용 | 관리 주체 |
|------|------|----------|
| 1 | 7개 트랙 → 하위 조직 트리(본부→팀→파트) → 조직장 지정 | 시스템관리자 |
| 2 | 사용자를 조직에 배치 (주 소속 + 겸직). 직급/직책 매핑 | 시스템관리자/HR |
| 3 | 역할 템플릿 적용 → 모듈 접근·데이터 범위·결재 권한 자동 부여 | 시스템관리자 |
| 4 | 모듈 카탈로그에서 필요한 모듈 선택 → 조직에 활성화 | 부서장 |
| 5 | 비주얼 워크플로우 엔진에서 크로스-트랙 프로세스 설계 | 경영기획/부서장 |

> Claude Code 가이드: `UserAssignment`(user_id, org_id, type[primary/concurrent/temporary/project], role_title, position_grade, is_org_head) + `RoleTemplate`(name, track_id, module_permissions:jsonb, data_scope, approval_limit) + `OrgModuleAssignment`(org_id, module_id, config:jsonb, status)

## 6. 권한 설계

### 6.1 7레벨 권한 계층

L0 Super Admin(시스템 전체) → L1 Track Admin(트랙 내) → L2 Org Admin(소속 조직) → L3 Team Lead(팀 내+하위 열람) → L4 Member(자기+팀 공유) → L5 Viewer(읽기 전용) → L6 External(파트너, 제한)

### 6.2 4층 권한 모델

Layer 1 시스템 권한(관리 메뉴) → Layer 2 모듈 권한(CRUD) → Layer 3 데이터 권한(자기/팀/부서/전사) → Layer 4 기능 권한(모듈 내 세부)

### 6.3 상속·위임

상위 조직 관리자 → 하위 데이터 접근 가능(설정으로 차단 가능). 부재 시 권한 위임(기간+범위+사유). 대리 결재(전결/후결).

> Claude Code 가이드: `PermissionDelegation`(delegator_id, delegatee_id, permission_scope:jsonb, start/end_date, reason, status)

### 7.9 주요 모듈 설계 보강

#### MKT-AUT 마케팅자동화 — 뉴스레터 발송 포함

MKT-AUT는 3가지 기능을 가진다: ① 여정 빌더(고객 여정 자동화) ② 트리거 메시지(행동 기반 자동 발송) ③ **뉴스레터 발송(대량 이메일 캠페인)**

**뉴스레터 발송 구조:**

```
[CRM-CDP 세그먼트]              [MKT-AUT 뉴스레터]             [전송 인프라]
 · VIP 고객                     · 드래그&드롭 에디터             · AWS SES
 · 30일 미방문                   · 템플릿 라이브러리              · 또는 SendGrid
 · 신규 가입 7일                 · 개인화 변수 {{이름}}           · IP 워밍업 자동
       │                        · A/B 테스트 (제목/본문)         · 바운스 자동 처리
       └──────→ 세그먼트 선택 ──→ · 예약 발송 / 즉시 발송     ──→ · 수신거부 자동
                                 · 발송 속도 조절 (throttle)
                                        │
                                        ▼
                                 [반응 수집 → CRM 자동 반영]
                                  · 오픈율 / 클릭률 / 전환율
                                  · 링크별 클릭 히트맵
                                  · 전환 추적 (UTM → MKT-ATR)
                                  · 수신거부 → CRM 동의 상태 즉시 반영
                                  · 고객 프로필에 행동 이력 기록
```

**자체 구축 vs 외부 위탁:**

| 구분 | 자체 (MKT-AUT) | 외부 위탁 (AWS SES) |
|------|---------------|-------------------|
| 구독자 관리 | ✅ CRM-CDP | — |
| 세그먼테이션 | ✅ CRM-CDP | — |
| 이메일 에디터 | ✅ 드래그&드롭 | — |
| 템플릿 | ✅ 라이브러리 | — |
| 개인화·A/B | ✅ | — |
| 반응 분석 | ✅ MKT-DTA/ATR 연동 | — |
| **전송 인프라** | — | ✅ **전송만 맡김** |
| 바운스/수신거부 | — | ✅ 자동 처리 |

> 스티비 같은 외부 뉴스레터 서비스 불필요. CRM 데이터를 외부에 넘기지 않으면서, 세그먼트→발송→반응→CRM 업데이트가 **플랫폼을 벗어나지 않고** 원스톱 처리.

## 8. 워크플로우 엔진·업무 도구 체계

### 8.1 3계층

| 계층 | 세팅 주체 | 범위 | 성격 |
|------|----------|------|------|
| **전사 워크플로우** | 경영기획/시스템관리자 | 전 부서 | 강제 — 표준 프로세스 |
| **부서별 워크플로우** | 부서장/팀장 | 해당 부서 | 자율 — 부서 특화 |
| **업무 도구** | 부서장(선택) / 개인(자동) | 부서원 / 본인 | 선택·자유 |

### 8.2 전사 워크플로우 (10종 표준)

| 워크플로우 | 핵심 흐름 |
|----------|----------|
| 전자결재 | 기안→팀장→(금액별)→본부장→CFO/CEO |
| 채용 | 부서요청→HR→공고→면접→오퍼→입사 |
| 온보딩 | (병렬) 좌석+장비+계정+교육+멘토 → 완료 |
| 퇴직 | 인수인계→반납→계정해지→정산 |
| 구매 | 견적비교→(금액별)→발주→입고→정산 |
| 예산 | 부서→재무→경영진→배정→집행 |
| 평가 | 목표→체크인→자기평가→다면→보정→확정 |
| 보상 | 평가→연봉조정→성과급→승인→지급 |
| 신제품 출시 | 기획→R&D→디자인→생산→물류→마케팅→영업 |
| 컴플라이언스 | 규제확인→영향분석→조치→실행→감사 |

> Claude Code 가이드: `WorkflowTemplate`(scope[company/department], trigger_type, steps[], cross_track, sla_hours, owner_org_id) → `WorkflowStep`(type[task/approval/condition/parallel_gate/timer/api_call], assignee_rule, timeout, escalation, next_steps[]) → `WorkflowInstance`(current_step, status, step_history[])

### 8.3 부서별 워크플로우 예시

마케팅=캠페인 실행·콘텐츠 발행, 영업=딜 프로세스, 개발=스프린트·장애 대응, 디자인=디자인 요청→납품, 생산=생산 실행·불량 처리, 인사=교육 운영, 재무=월 마감

부서장이 템플릿 라이브러리에서 선택하거나 커스텀 설계(드래그&드롭). 전사 워크플로우에 부서 단계를 삽입 가능.

### 8.4 부서 공통 도구

부서장이 카탈로그에서 골라 활성화:

| 카테고리 | 도구 |
|---------|------|
| 소통 | 게시판(COM-BRD), 메신저(COM-MSG) |
| 협업 | 프로젝트(COM-PRJ), 문서(COM-DOC), 캘린더(COM-CAL) |
| 업무 | **업무 캘린더(COM-WCL)** — 주간·월간·분기·연간, 팀→부문→전사 상향 집계 |
| 결재 | 전자결재(COM-APR) |
| 지식 | 위키, LMS |
| 분석 | 리포트(COM-RPT) |
| AI | AI어시스턴트(COM-AI) |

### 8.5 업무 캘린더 (COM-WCL) — 상향 집계 구조

```
팀원 업무 → [팀 캘린더] → 팀장 ★선택 → [부문 캘린더] → 본부장 ★★선택 → [전사 캘린더]
```

| | 팀 | 부문 | 전사 |
|---|---|---|---|
| 주간 | 팀원 업무 전체 | 팀 ★주요 업무 집계 | 부문 ★★핵심 집계 |
| 월간 | 팀 목표·계획 | 팀 월간 집계 | 부문 월간 집계 |
| 분기 | 팀 핵심 과제 | 팀 분기 집계 | 부문 분기 집계 |
| 연간 | 팀 연간 계획 | 팀 연간 집계 | 부문 연간 집계 |

**상향 규칙:** 팀장이 ★표시 → 부문에 노출. 본부장이 ★★표시 → 전사에 노출. 예산 초과·SLA 위반·마감 지연은 **시스템이 자동 상향**.

**연동:** COM-CAL(일정)과 통합 표시, COM-PRJ 마일스톤 자동 반영, HR-WRK(업무 배정) 자동 연결, HR-EVL+(분기 실적 참조), STR-KPI(전사 목표 마일스톤), COM-APR(결재 마감 연동)

```

├──────────────────────────────────────────────────────────────┤
│  월    화    수    목    금                                   │

│  □ 데이터  □ 콘텐츠         ★ 예산   □ 주간                  │
```

## 9. 평가·보상 체계

### 9.1 설계 원칙

성과(What)+가치(How) 동등, 다면 평가, 상시 피드백→평가 연결, 투명한 기준, 성장 연결(처벌 아님)

### 9.2 주기

상시(피드백) → 분기(체크인: 목표+가치+성장) → 연간(종합: 성과+역량+다면) → 보상(연봉+인센티브+승진) → 성장(IDP)

### 9.3 성과 평가 — 캐스케이딩

기업 미션 → 전사 목표 → 트랙 → 부서 → 팀 → 개인. 각 단계에서 정렬도(Alignment Score) 자동 분석.

### 9.4 역량 평가 — 핵심가치 기반

Culture Engine의 `CoreValue.behavioral_indicators`가 평가 항목으로 자동 생성. 5단계 척도(needs_improvement ~ exceptional) + 근거 + 피드백 참조.

### 9.5 다면 평가

| 유형 | 평가자 | 비중 | 항목 |
|------|--------|------|------|
| 상향 | 직속 상사 | 40% | 성과, 역량, 잠재력 |
| 동료 | 협업 동료 3~5명 | 30% | 협업, 소통, 가치 실천 |
| 하향 | 직속 팀원 | 20% | 리더십, 코칭 (관리자만) |
| 외부 | 고객/파트너 | 10% | 전문성, 신뢰 (해당 직군) |

### 9.6 피드백·인정

**상시 피드백**: 핵심가치 태그 + 관찰 + 영향 + 제안. 프라이버시 설정(비공개/상사공유/공개칭찬).
**넛지**: 프로젝트 완료 시, 1:1 전, 분기말 2주 전, 월요일 아침.
**인정**: Spot Award(수시 포인트), Monthly MVP(동료 투표), Value Champion(분기, AI 추천), Innovation Award(연간).

### 9.7 보상

고정(기본급+직책수당+역량수당) + 변동(성과 보너스: 개인60%+팀25%+전사15% × 등급 배수 S=2.0/A=1.5/B=1.0/C=0.5/D=0.0) + 비금전(포인트, 복지몰, 추가 휴가). 핵심가치 실천 우수자 추가 가산.

> Claude Code 가이드: `PerformanceGoal`(goal_type, metric_type, target/actual_value, weight, core_value_alignment, parent_goal_id) + `CompetencyEvaluation`(evaluator_type, items[core_value+score+evidence]) + `FeedbackEntry`(type, core_value_id, observation, impact, visibility) + `PromotionCriteria`(min_tenure, min_performance, required_certs, leadership_assessment)

## 10. 통합 CRM — 3층 구조

### 10.1 핵심 개념

소비자는 하나의 통합 ID로 그룹사 모든 서비스에 접속 (구글 계정 모델).

### 10.2 3개 Layer

**Layer 1: 통합 고객 (Golden Record)** — 지주사 관리
통합 ID, 이메일/폰(unique), 멤버십 등급(basic~vip), 통합 LTV, 통합 포인트, 동의 관리(BU별×채널별×목적별), AI 페르소나

**Layer 2: 사업회사별 프로필** — 각 BU 관리
로컬 ID, BU별 등급/LTV/구매액, 선호 채널, engagement 점수, 이탈 위험 점수

**Layer 3: 접점 데이터 (Touchpoint)** — 11개 채널
web, app, 오프라인, 콜센터, 이메일, SMS, 푸시, 챗봇, 소셜, 키오스크, 파트너. 접점 유형(방문/구매/문의/불만/캠페인응답/리뷰/추천)

### 10.3 Identity Resolution

deterministic(이메일/폰: confidence 1.0) + probabilistic(디바이스ID: 0.9, POS카드: 0.8). 카카오/네이버 소셜 ID 연동.

### 10.4 지주사 vs 사업회사 역할

| 기능 | 지주사 | 사업회사 |
|------|--------|---------|
| 고객 ID | 통합 발급, Resolution | 로컬 매핑 |
| 세그먼트 | 전사(VIP, 이탈위험) | 사업 특화 |
| 캠페인 | 크로스셀링, 통합 멤버십 | 개별 프로모션 |
| 분석 | 전사 고객 인사이트 | 서비스별 퍼널 |
| 개인정보 | 통합 동의 관리 | 범위 내 활용 |
| 멤버십 | 통합 등급/포인트 | 서비스별 혜택 |

### 10.5 소비자 경험

통합 앱/포털: SSO(카카오/구글/애플) → 내 정보 + 서비스 허브(BU별 딥링크) + 통합 이용 내역 + 혜택/이벤트(개인화 추천) + 통합 고객센터(AI 챗봇)

### 10.6 개인정보보호

목적 외 사용 금지, 최소 수집, 보관 기한 필수. 동의는 목적별+BU별 세분화, 실시간 철회. 데이터 주체 권리(열람/정정/삭제/이동) 자동화. AES-256, 가명처리, 전 조회 기록.

## 11. 대외 마케팅 커뮤니케이션

### 11.1 매체 분류 — 3분류

**Paid**: 디지털 검색(네이버SA/구글/카카오), 디스플레이(GDN/DA/프로그래매틱), 소셜(메타/틱톡/X/링크드인), 동영상(유튜브/OTT), 커머스(쿠팡/네이버쇼핑), TV, 라디오, 옥외(OOH/DOOH), 인쇄, 인플루언서
**Owned**: 웹사이트, 앱, 이메일, SMS, 푸시, 카카오톡 채널, 블로그, SNS 공식계정
**Earned**: 언론 보도, 소셜 버즈, 리뷰/평점, 바이럴/UGC, 검색 트렌드

> Claude Code 가이드: `MediaChannel`(ownership[paid/owned/earned], category, platform, api_type, sync_frequency, billing_model, available_metrics[], attribution_support, vendor_id, status)

### 11.2 캠페인 라이프사이클 — 10단계

전략 수립 → 기획 → 제작 → 검수/승인 → 집행 → 모니터링 → 최적화 → 종료/정산 → 분석/보고 → 학습/아카이브

> Claude Code 가이드: `Campaign`(business_unit_id, code[BU-연도-분기-유형], campaign_type[10종], target_audience:AudienceDefinition, primary_kpi, budget, media_plan[], creatives[], actual_spend, approval_status, holding_group_id). `AudienceDefinition`(데모+사이코+행동+CRM세그먼트+유사타겟). `MediaPlanEntry`(계획 vs 실적, 효율 자동계산). `Creative`(유형, A/B, 심의, 성과).

### 11.3 반응 데이터 3개 레이어

**Layer 1: 매체 반응** — API 자동 수집. 노출/도달/클릭/영상조회/참여/비용 + 효율 자동 계산(CPM/CPC/CTR). 오디언스 분석. 원본(raw_data) 보존.

**Layer 2: 전환 반응** — UTM/SDK 기반. 전환 유형(가입/장바구니/구매/설치/전화/양식/다운로드/예약/매장방문). 비식별 사용자 → 식별 시 UnifiedCustomer 연결. 어트리뷰션 모델별 기여도(0.0~1.0). 전환 경로(터치포인트 시퀀스).

**Layer 3: 감성/인식** — 소셜 리스닝 + 미디어 모니터링. AI 감성 분석(-1.0~+1.0), 감정(joy/trust/anger 등), 토픽 추출, 의도(칭찬/불만/질문/제안), 영향력(팔로워/참여수/도달), 대응 필요 여부.

### 11.4 매체 용어 통합

### 13.4 모듈 의존성 — 7대 폭발 클러스터

~110개 모듈 간 의존성에서 경우의 수가 집중되는 7개 지점:

| # | 클러스터 | 폭발 원인 |
|---|---------|----------|
| ① | 인사→업무→평가→보상 | 직급×직군×부서×평가유형×보상정책 = 수백 |
| ② | 결재·예산·정산 삼각형 | 결재유형×금액구간×부서×예산상태×예외 = 수천 |
| ③ | 캠페인→매체→어트리뷰션 | 매체×API버전×어트리뷰션모델×캠페인유형×BU = 무한 |
| ④ | 고객 ID 통합 | BU×채널×동의유형×매칭방법×멤버십 = 기하급수 |
| ⑤ | 생산→품질→물류 | 품목×검사결과×불량유형×공급업체×리드타임 = 제조 특유 |
| ⑥ | 워크플로우 엔진 자체 | 전사WF×부서WF×삽입×예외×시간 = 설계 핵심 |
| ⑦ | 권한×데이터 범위 교차 | 역할×겸직×위임×트랙간접근×시간 = 차원 급증 |

**가장 밀도 높은 교차:** T1↔T6(인사/재무↔전직원 대시보드), T2↔T3(수주→생산→물류), T1↔T2(예산/결재↔사업)

### 13.5 8대 설계 원칙

| # | 원칙 | 핵심 |
|---|------|------|
| 1 | **Config-Driven** | 경우의 수를 if/else가 아닌 설정 테이블로 흡수. 필수: 워크플로우·권한·평가·매체매핑 |
| 2 | **Contract-First Handoff** | 크로스-트랙 핸드오프에 조건·스키마·실패처리 명시. 미정의 데이터 전달 금지 |
| 3 | **Rule Engine** | 결재·평가·정산의 중첩 조건을 전용 엔진으로. 조건 트리(AND/OR)+액션+priority. 관리자가 규칙 추가 시 즉시 반영 |
| 4 | **Event Sourcing** | 모든 상태 변경을 이벤트로 기록. "왜 이렇게 됐는가" 추적(어떤 규칙, 어떤 워크플로우) |
| 5 | **Schema Registry** | 모듈 간 데이터 흐름 스키마 중앙 관리. 버전·호환성·마이그레이션. 매체 API 변경 시 Adapter+자동 감지 |
| 6 | **Tenant Override** | System Default → Tenant Override → Org Override 3층. 오버라이드 제약(min_approvers 등)으로 무결성 보호 |
| 7 | **Graceful Degradation** | 동기 최소화, 비동기 이벤트 기반, DLQ 재처리. required+sync 조합 최소화 |
| 8 | **Progressive Disclosure** | Level A(5분 Quick Setup: 업종 프리셋) → Level B(1시간 Customize) → Level C(전문가 Advanced) |

### 15.1 개념

### 15.2 4계층 대시보드

| Level | 대상 | 핵심 화면 |

같은 사람이 여러 레벨을 오간다 (본부장=L2+L4, 팀장=L3+L4).

## 19. End-to-End 데이터 흐름도

### 19.1 수주부터 정산까지

```
[영업] SAL-PIP 기회 등록
  → SAL-QOT 견적 작성 → COM-APR 견적 승인
  → SAL-ORD 수주 확정
      │
      ├──→ [생산] PRD-PLN 생산계획 자동 생성
      │         → PRD-PRC 자재 발주 (PRD-INV 재고 체크)
      │         → PRD-MES 공정 투입 → PRD-QC 품질 검사
      │         → LOG-WMS 완제품 입고
      │
      ├──→ [물류] LOG-TMS 배송 → 고객 수령 확인
      │
      ├──→ [재무] FIN-AR 매출 인식 → 청구 → 수금
      │         FIN-AP 자재 대금 → FIN-GL 전표
      │
      └──→ [CRM] CRM-CST 고객 이력 업데이트 → CRM-CX 만족도 조사
```

**관련 트랙:** T2(영업)→T3(생산·물류)→T1(재무)→T2(CRM). 4개 트랙이 하나의 흐름으로 연결.

### 19.2 채용부터 퇴직까지

```
[채용] HR-REC 공고 → 지원 → AI 스크리닝 → 면접 → 오퍼 → 입사 확정
  → [온보딩] (병렬) 좌석+장비(총무) + 계정(IT) + 교육(HR) + 멘토(팀장)
  → [배치] HR-ORG 조직 배치 → 역할·권한 자동 부여 → 모듈 접근
  → [업무] HR-WRK 업무 배정 → COM-PRJ 프로젝트 참여 → 시수 축적
  → [성장] HR-EDU 교육 → HR-FBK 상시 피드백 → HR-EVL+ 분기 평가
  → [보상] HR-REW 연봉 조정 + HR-INC 성과급 → HR-PAY 급여
  → [퇴직] 퇴직 신청 → 인수인계 → 데이터 이전 → 장비 반납 → 계정 해지 → 정산
```

**전 과정이 MY-HOME에서 개인 기준으로 보임.** 직원은 자기 채용 상태, 온보딩 진행률, 업무, 평가, 급여를 하나의 대시보드에서 확인.

### 19.3 캠페인 기획부터 ROI 보고까지

```
[전략] MKT-STR 연간 전략 → 분기 캠페인 계획
  → [기획] MKT-CMP 캠페인 생성 (타겟·KPI·예산·기간)
  → [제작] MKT-CRT 크리에이티브 요청 → 시안 → 검수 → A/B테스트
  → [집행] MKT-MDI 매체 플랜 → MKT-PFM 광고 세팅 → 라이브
  → [모니터링] MKT-DTA 데이터 자동 수집 → 실시간 대시보드
  → [최적화] AI 이상 감지 → 소재 교체 / 예산 재배분 / 타겟 조정
  → [종료] MKT-OPS 매체비 정산 → FIN-AP 대금 지급
  → [분석] MKT-ATR 어트리뷰션 → MKT-MMM 미디어믹스 → ROI 보고서
  → [학습] 성과 아카이브 → 다음 캠페인 AI 추천에 반영
```

### 19.4 기안부터 정산까지

```
[기안] 직원이 COM-APR에서 양식 선택 → 내용 작성 (AI 초안)
  → [AI 검증] 가치 정합성 체크 + 예산 잔액 확인
  → [결재] Rule Engine이 경로 자동 결정 (금액·유형·부서)
      → 팀장 → (분기) → 본부장 → CFO
  → [승인] 후속 액션 자동 트리거
      → 구매 기안이면 → PRD-PRC 발주
      → 출장 기안이면 → COM-CAL 일정 + 총무 숙박
      → 매체비면 → MKT-OPS 집행
  → [집행] 실제 사용
  → [정산] FIN-AP 전표 자동 생성 → FIN-GL 원장 반영
  → [추적] MY-APR에서 전 과정 추적 가능
```

| **방해 금지** | 개인 설정 (22:00~08:00). 긴급만 관통 |
| **스마트 채널** | AI가 사용자 패턴 분석 → 모바일 활성 시간엔 푸시, 데스크톱 시간엔 인앱 |
| **읽음 추적** | 인앱에서 읽었으면 이메일 발송 취소 |
| **구독 관리** | 알림 유형별 켜기/끄기. 최소 필수 알림(결재·보안)은 끌 수 없음 |

## 26. 경쟁 포지셔닝

### 26.1 기존 솔루션 vs EUS

| | SAP/Oracle | Workday | Monday/Notion | Slack+Salesforce | **WIO EUS** |
|---|---|---|---|---|---|
| **접근** | 기능 중심 | HR 특화 | 협업 도구 | 조합형 | **조직 중심+모듈 조합** |
| 커스터마이징 | 컨설팅 필수 | 제한적 | 높음(표면) | 각각 설정 | **비주얼 세팅(레고)** |
| 문화 내재화 | 없음 | 일부 | 없음 | 없음 | **Culture Engine** |
| AI 통합 | 별도 모듈 | 제한적 | 기본 | 각 제품별 | **전 모듈 AI 80%** |
| 크로스-기능 | 모듈 간 연동 | HR 내부 | 프로젝트 내 | 제품 간 연동 | **트랙 간 자동 워크플로우** |
| 도입 시간 | 12~24개월 | 6~12개월 | 1주 | 각각 도입 | **5분(프리셋)~3개월** |
| 가격 | 사용자당 $150+ | 사용자당 $100+ | 사용자당 $10~25 | 합산 $50+ | **사용자당 $10~30** |

### 26.2 WIO EUS만의 차별점 3가지

**① 조직이 먼저, 기능은 그 다음.**
SAP는 "재무 모듈을 사고 → 조직을 맞추는" 방식. EUS는 "조직을 세팅하고 → 필요한 모듈을 끼우는" 방식. 조직 구조가 바뀌면 모듈도 자동으로 재배치.

**② Culture Engine — 철학이 시스템에 녹는다.**
기업의 미션·가치가 양식·결재·평가·피드백에 구조적으로 내장. 다른 어떤 ERP도 이걸 하지 않음.

**③ AI가 80%를 채운다.**
전 모듈에 AI가 내장. 입력·분석·추천·생성·검증·자연어 6가지 유형. 별도 AI 모듈을 사는 게 아니라 시스템 자체가 AI.

## 27. 추가 설계 보강

### 27.1 오프보딩(퇴직) 프로세스

```
퇴직 신청 → HR 확인
  → (병렬) 인수인계 계획 수립 + 잔여 휴가 정산
  → 인수인계 실행 (후임자에게 프로젝트·문서·고객 이전)
  → 장비 반납 (전산 체크리스트)
  → 계정 처리: D-Day 18:00 접근 차단 → 30일 데이터 보관 → 삭제/아카이브
  → 지식 이전: 개인 위키·문서 → 팀 공유 폴더로 자동 이전
  → 퇴직 정산 (급여·퇴직금·미정산 경비)
  → 퇴직 면담 기록 → HR 아카이브
```

### 27.2 다국어/다통화/다시간대

| 기능 | 설명 |
|------|------|
| 다국어 | UI 언어 전환 (한/영/중/일). 콘텐츠는 원본 언어 유지 + AI 번역 |
| 다통화 | 기준 통화 설정 + 실시간 환율 → 자동 변환. 법인별 다른 기준 통화 |
| 다시간대 | 사용자별 시간대 설정. 업무 캘린더 겹침 뷰에서 시간대 자동 조정 |
| 현지 법규 | 국가별 세법·노동법 모듈 (4대보험 → 현지 보험 체계로 전환) |

### 27.3 보고서/BI 고도화

| 기능 | 설명 |
|------|------|
| 표준 보고서 | 부서별 자동 생성 (주간/월간/분기). AI가 초안 → 팀장 확인 |
| 셀프서비스 BI | 직원이 직접 대시보드 구성 (드래그&드롭 위젯, 필터, 차트 유형) |
| AI 리포트 | "이번 달 마케팅 성과 보고서 만들어줘" → AI가 데이터 수집+분석+시각화+초안 |
| 스케줄 발송 | 보고서를 매주 월요일 09시에 자동 생성 → 메일/메신저 발송 |
| 권한 기반 | 같은 보고서도 직급에 따라 상세도가 다름 (팀원=팀 데이터, 본부장=부문 전체) |

### 27.4 통합 검색 엔진

| 검색 대상 | 인덱싱 |
|----------|--------|
| 사람 | 이름, 부서, 직급, 역량 태그 |
| 문서 | 제목, 본문, 작성자, 태그 (COM-DOC, 위키) |
| 결재 | 기안명, 기안자, 상태, 금액 |
| 프로젝트 | 프로젝트명, 멤버, 태스크 |
