# WIO Glossary — 용어 및 체계 사전

> 버전: v1.0
> 날짜: 2026-03-30
> 목적: WIO 개발·커뮤니케이션·외부 설명에서 용어를 통일한다
> 기준: EUS v2.0 §1.4 개정 반영

---

## 1. 7계층 구조

WIO의 모든 개념은 7개 계층으로 정리된다. 위에서 아래로 갈수록 구체적이다.

```
L1 제품     WIO = Orbi(조직) + Myverse(개인)
L2 트랙     7개 부문. C-Level이 관장하는 영역
L3 조직     본부 → 팀 → 파트 → 셀. 사람이 소속되는 곳
L4 서비스   사용자가 보는 기능 묶음 (12개)
L5 모듈     시스템 코드 단위. 레고 블록 (~110개)
L6 툴      모듈 안의 기능적 UI 요소
L7 엔진     전 모듈이 공유하는 인프라
```

**L1~L3 = 조직 구조 (누가, 어디서)**
**L4~L6 = 기능 구조 (무엇을, 어떤 도구로)**
**L7 = 인프라 (어떻게 작동하는가)**

---

## 2. 계층별 상세 정의

### L1. 제품 (Product)

| 용어 | 정의 | 비유 |
|------|------|------|
| **WIO** | Work In One. AI Business OS 브랜드명. "입력을 없앤다. AI가 80% 채운다. 사람은 20%의 판단" | 자동차 브랜드명 |
| **Orbi** | 조직용 웹 애플리케이션. 인공지능 모듈러 ERP. 회사가 일하는 곳 | 자동차의 "세단" 라인업 |
| **Myverse** | 개인용 모바일 앱. Personal Black Box(기록) + 소셜 관리(Hootsuite) + Orbi 접속(일하기) | 자동차의 "개인 키" — 키로 차에 접근 |

**관계:** Myverse 사용자가 Orbi에 접속해서 일한다. 인증은 공유(Supabase Auth), 데이터는 분리(개인 데이터 ≠ 조직 데이터).

---

### L2. 트랙 (Track)

C-Level이 관장하는 부문 단위. 7개.

| 트랙 | 영역 | 관장 C-Level | 대상 조직 예시 |
|------|------|-------------|--------------|
| **T1 운영·관리** | 인사, 재무, 회계, 법무, 감사, 총무, 경영기획 | CHO(인사) + CFO(재무) | 인사팀, 재무팀, 법무팀, 경영기획팀 |
| **T2 사업** | 마케팅, 영업, CRM, 사업개발, 고객관리 | CMO(마케팅) + CBO(사업) | 마케팅본부, 영업팀, CS팀 |
| **T3 생산** | 생산계획, 공정, 품질, 설비, 구매, 물류 | COO(운영) | 생산팀, 품질관리팀, 물류팀 |
| **T4 지원** | R&D, 개발, 디자인, 데이터 | CTO(기술) | 개발팀, 디자인팀, R&D센터 |
| **T5 파트너** | 협력사, 프리랜서, 컨설팅 | — (파트너 관리 부서) | 외부 파트너 조직 |
| **T6 공통** | 메신저, 결재, 캘린더, 대시보드, 피드백 | — (전 직원 사용) | 모든 조직 |
| **T7 시스템** | 사용자, 권한, 조직, 모듈, 보안, 감사 | CIO(정보시스템) | IT팀, 전산팀 |

**비유:** 건물의 층. T1은 경영 지원층, T2는 영업·마케팅층, T3은 공장층.

**역할:**
- 트랙은 모듈이 소속되는 카테고리다
- 트랙별로 Track Admin(L1 권한)이 관리한다
- 업종 프리셋은 트랙 단위로 활성/비활성한다

---

### L3. 조직 (Organization)

트랙 안의 실제 부서. 사람이 소속되는 곳.

| 레벨 | 예시 | 설명 |
|------|------|------|
| **본부** | 마케팅본부, 영업본부 | 최상위 조직. 본부장이 관장 |
| **팀** | 마케팅전략팀, 캠페인팀 | 본부 아래. 팀장이 관장 |
| **파트** | 퍼포먼스파트, 브랜드파트 | 팀 아래. 파트장이 관장 |
| **셀** | 검색광고셀, SNS셀 | 파트 아래. 최소 단위 |

**비유:** 건물 층(트랙) 안의 방. 방마다 사람이 있고, 방마다 다른 가구(모듈)가 놓인다.

**특수 조직 유형:**
- **TF/프로젝트팀** — 가상 조직. 트랙과 무관하게 생성. 참여자는 원 소속 유지
- **겸직** — 1인이 주 소속 + 추가 소속을 동시에 가짐
- **파견** — 원 소속 유지하면서 다른 조직에서 근무

---

### L4. 서비스 (Service) ← **신규 정의**

사용자가 보는 기능 묶음. Flow의 "프로젝트 관리", "업무 관리" 같은 것.

**서비스는 사용자 언어다.** 사용자는 "Talk을 쓴다"고 말하지, "COM-MSG 모듈의 conversations 테이블에 접근한다"고 말하지 않는다.

| 서비스 | 한 줄 설명 | 비유 |
|--------|----------|------|
| **Talk** | 대화하고 소통하기 | 카카오톡 + 슬랙 |
| **Project** | 프로젝트를 관리하기 | Asana + Monday |
| **Task** | 업무 상태를 추적하기 | Jira + Trello |
| **Board** | 공지·게시·위키 | 네이버 카페 + Notion |
| **Approval** | 기안하고 결재받기 | 그룹웨어 전자결재 |
| **Workflow** | 업무 흐름을 설계하기 | Zapier + n8n |
| **People** | 사람과 조직을 관리하기 | Workday |
| **Goal** | 목표를 세우고 평가받기 | Lattice + 15Five |
| **Finance** | 돈을 관리하기 | SAP FI |
| **Timesheet** | 시간을 기록하고 청구하기 | Toggl + Harvest |
| **Marketing** | 캠페인과 매체를 관리하기 | HubSpot + GA4 |
| **CRM** | 고객을 관리하기 | Salesforce |

**+ 관통 서비스:**
| 서비스 | 설명 |
|--------|------|
| **AI assistant** | 전 서비스에 내장된 AI. 자연어로 질의, 초안 생성, 코칭 |
| **Culture engine** | 기업 철학을 양식·결재·평가에 녹이는 엔진 |

**서비스 vs 모듈의 관계:**
- 서비스 1개 = 모듈 N개의 사용자 뷰
- 서비스는 사이드바에 표시되는 이름
- 모듈은 관리자가 카탈로그에서 배정하는 코드

---

### L5. 모듈 (Module)

시스템 코드 단위. ~110개. 카탈로그에서 골라 조직에 배정하는 "레고 블록".

| 속성 | 설명 |
|------|------|
| **코드** | HR-REC, MKT-CMP, FIN-GL 등. 트랙-기능 조합 |
| **소속 트랙** | T1~T7 + Holding 중 하나 |
| **독립성** | 각 모듈은 독립 배포·업데이트 가능 (마이크로서비스) |
| **배정** | 관리자가 카탈로그에서 선택 → 조직에 드래그&드롭 |
| **설정** | 모듈별 접근 권한, 데이터 범위, 알림 설정 가능 |

**비유:** 방(조직)에 놓는 가구. 마케팅팀 방에는 캠페인 가구(MKT-CMP), 매체 가구(MKT-MDI), 퍼포먼스 가구(MKT-PFM)를 놓는다. 재무팀 방에는 원장 가구(FIN-GL), 예산 가구(FIN-BUD)를 놓는다.

**모듈은 개발자 언어다.** Claude Code에게 "MKT-CMP 모듈의 campaigns 테이블에 status 필드를 추가해"라고 지시한다.

---

### L6. 툴 (Tool)

모듈 안의 기능적 UI 요소. 여러 모듈에서 재사용될 수 있다.

| 툴 | 사용되는 서비스/모듈 | 설명 |
|----|-------------------|------|
| **간트차트** | Project, Task | 일정·의존성을 시각화하는 차트 |
| **칸반보드** | Project, Task | To Do → In Progress → Done 보드 |
| **캘린더 뷰** | Task, People | 일정을 달력 형태로 표시 |
| **리스트 뷰** | Task, People, Finance | 데이터를 표 형태로 표시 |
| **에디터** | Board, Approval | 문서·기안서를 작성하는 WYSIWYG |
| **차트/그래프** | Finance, Marketing, Goal | 데이터를 시각화하는 그래프 |
| **통합 검색** | 전 서비스 | 사람·문서·결재·프로젝트·메시지를 한 번에 검색 |
| **AI 브리핑** | Home 대시보드 | 오늘 할 일·일정·결재를 AI가 요약 |
| **비주얼 빌더** | Workflow, 설정 | 노드를 배치하고 선으로 연결하는 플로우 설계 도구 |
| **트리 빌더** | People, 설정 | 조직도를 드래그로 편집하는 트리 에디터 |

**비유:** 방(조직)의 가구(모듈) 위에 놓인 문구·장비. 간트차트는 프로젝트 가구 위에도, 업무 가구 위에도 놓을 수 있다.

---

### L7. 엔진 (Engine)

전 모듈이 공유하는 시스템 인프라. 사용자 눈에 보이지 않지만, 이것 없이는 모듈이 작동하지 않는다.

| 엔진 | 역할 | 현재 상태 |
|------|------|----------|
| **RBAC** | 6단계 권한 체계 (super_admin→guest). 모듈 접근, 데이터 범위, 기능 권한 | ✅ 실작동 |
| **워크플로우 엔진** | 워크플로우 실행. startWorkflow, advanceStep, checkSLA | ✅ 실작동 |
| **Culture Engine** | 기업 철학 내재화. evaluateCultureFit, getCultureHealth | ✅ 실작동 |
| **Rule Engine** | 결재·평가·정산의 중첩 조건 처리. 조건 트리(AND/OR) + 액션 | 설계 완료, 구현 대기 |
| **Event Bus** | 모듈 간 이벤트 전달. 인력 배치 → 권한+채널+결재 자동 연동 | 설계 완료, 구현 대기 |
| **Config Framework** | 경우의 수를 if/else가 아닌 설정 테이블로 흡수 | 설계 완료, 구현 대기 |

**비유:** 건물의 전기·수도·인터넷 배관. 눈에 안 보이지만 없으면 방 안의 가구가 작동하지 않는다.

## 3. 관통 개념 — 특정 계층에 속하지 않고 전체를 관통하는 것

| 개념 | 정의 | 소속 | 비유 |
|------|------|------|------|
| **워크플로우** | 업무가 흘러가는 순서·규칙·조건. 비주얼 빌더로 설계 | L6 툴 + L7 엔진 | 건물의 복도와 동선 |
| **결재** | 워크플로우 중 승인 노드가 포함된 것. 회사의 공식 승인 | Approval 서비스 | 출입문의 잠금장치 |
| **권한** | 7레벨 × 4층. 누가 무엇을 볼 수 있고 할 수 있는지 | L7 엔진 (RBAC) | 출입 카드 |
| **양식** | 기업 철학이 내장된 공식 문서 서식 | Culture Engine | 표준 서류함 |
| **대시보드** | 역할별 데이터 시각화. 4계층 (전사/부문/부서/개인) | L6 툴 | 안내 전광판 |
| **AI** | 전 모듈에 내장. 6유형 (입력·분석·추천·생성·검증·자연어) | 관통 서비스 | 건물의 자동화 시스템 |

**워크플로우 vs 결재 구분:**
- 워크플로우 = 업무 흐름 전체 (시작→작업→조건분기→자동액션→종료)
- 결재 = 워크플로우 안에 "승인" 노드가 들어간 것
- 모든 결재는 워크플로우이지만, 모든 워크플로우가 결재는 아니다

## 5. 업종 프리셋별 서비스 구성

업종 프리셋을 선택하면, 해당 업종에 맞는 서비스만 사이드바에 표시된다.

| 프리셋 | 활성 서비스 | 비활성 |
|--------|-----------|--------|
| **에이전시** | Talk, Project, Task, Approval, Timesheet, Finance, Marketing, AI | People(축소), CRM(축소), Goal(축소) |
| **제조업** | Talk, Task, Approval, People, Finance, Workflow, AI + 생산 전용 | Marketing(비활성), Timesheet(축소) |
| **IT/SaaS** | Talk, Project, Task, Marketing, CRM, AI + 개발 전용 | Finance(축소), Timesheet(축소) |
| **유통** | Talk, Task, Approval, CRM, Finance, AI + 물류 전용 | Marketing(축소), Project(축소) |

**Progressive Disclosure 원칙:** 어떤 프리셋이든 첫 화면에는 3개만 보인다 — Home(오늘 할 일) + Talk(메시지) + Approval(결재).

## 6. 누가 무엇을 다루는가

| 역할 | 다루는 계층 | 하는 일 |
|------|-----------|--------|
| **사용자 (직원)** | L4 서비스, L6 툴 | 서비스를 통해 일한다. Talk으로 대화, Task로 업무 관리, Approval로 결재 |
| **관리자 (어드민)** | L2 트랙, L3 조직, L5 모듈 | 트랙에 조직을 만들고, 조직에 모듈을 배정하고, 권한을 설정한다 |
| **개발자 (Claude Code)** | L5 모듈, L6 툴, L7 엔진 | 모듈을 만들고, 툴을 구현하고, 엔진을 작동시킨다 |
| **경영진 (C-Level)** | L2 트랙, 대시보드 | 자기 트랙의 대시보드에서 KPI를 보고 의사결정한다 |
| **텐원 (설계자)** | 전체 | 체계를 설계하고 방향을 잡는다 |

## 7. 세팅 순서 (개정)

```
Step 1. 트랙 생성 (7개 중 필요한 것 활성화)        → 시스템관리자
Step 2. 조직 배치 (본부→팀→파트 트리 구성)          → 시스템관리자
Step 3. 인력 배치 (사용자를 조직에 배정, 직급/직책)   → 시스템관리자/HR
Step 4. 권한 설정 (역할 템플릿 → 자동 부여)          → 시스템관리자
Step 5. 서비스 활성화 (업종 프리셋 또는 수동 선택)     → 시스템관리자
Step 6. 모듈 배정 (서비스 안의 세부 모듈 조합)        → 부서장
Step 7. 워크플로우 설계 (전사 결재 + 부서별 업무 흐름) → 경영기획/부서장
Step 8. 대시보드 자동 제공 (역할별 위젯 구성)         → 자동
```

**변경점:** 기존 Step 5가 "모듈 배정"이었으나, "서비스 활성화"가 먼저 오고, 그 안의 세부 모듈 조합이 Step 6으로 분리됨. 사용자는 서비스를 보고, 관리자는 서비스 안의 모듈을 조합한다.

## 8. 용어 사용 규칙

### 대화에서

| 상황 | 올바른 표현 | 잘못된 표현 |
|------|-----------|-----------|
| 사용자에게 설명할 때 | "Talk 서비스에서 메시지를 보내세요" | "COM-MSG 모듈을 사용하세요" |
| 관리자에게 설명할 때 | "마케팅팀에 캠페인 모듈을 배정하세요" | "MKT-CMP를 org_module_assignments에 INSERT하세요" |
| Claude Code에게 지시할 때 | "MKT-CMP 모듈의 campaigns 테이블에 status 필드 추가해" | "마케팅 서비스 고쳐줘" |
| 소개 자료에서 | "Orbi는 12개 서비스로 구성된 AI 모듈러 ERP입니다" | "Orbi는 110개 모듈로 구성됩니다" |

### 문서에서

- **기획서/소개서** → 서비스 단위로 설명 (Talk, Project, Task...)
- **개발 문서** → 모듈 코드로 설명 (COM-MSG, HR-WRK...)
- **설정 가이드** → 트랙 + 조직 + 모듈 단위로 설명
- **API 문서** → 모듈 코드 + 테이블명으로 설명
